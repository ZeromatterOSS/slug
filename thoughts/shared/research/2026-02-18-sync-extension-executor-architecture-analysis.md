# Research: Sync Extension Executor Architecture Analysis

**Date**: 2026-02-18T19:43:25-08:00
**Researcher**: wgray
**Git Commit**: 71c5e7af83dccfb2f14669ea16ca86ffcdf93bbf
**Branch**: stable-rust-migration
**Repository**: slug

## Research Question

Compare the `sync_extension_executor`'s implementation to Bazel's .bzl file loading, analyze whether Buck2 has anything like bzlmod's `repository_rule`s, note inconsistencies, and analyze potential alternatives to the current implementation.

## Summary

Slug's `SyncExtensionExecutor` exists to solve a **bootstrapping problem**: cell aliases from `use_repo()` must be registered before DICE starts, but module extension execution normally requires DICE to load `.bzl` files. This creates a chicken-and-egg situation that Bazel solves entirely differently: through **deterministic pre-computed canonical names** and a **lockfile cache** within its Skyframe DAG. Buck2 has **no equivalent to bzlmod or repository_rule()** at all—its cell system is static, declared in `.buckconfig` files.

The key architectural inconsistency is that Slug's sync executor is a **parallel Starlark evaluation environment** with a subset of globals, separate path resolution, no cell awareness, and a `Mutex<HashMap>` module cache—diverging from the normal DICE-backed path in multiple dimensions. This approach works for root-project extensions but will fail for any extension that `load()`s from an external cell.

## Detailed Findings

### 1. How Bazel Handles .bzl Loading for Module Extensions and Repository Rules

Bazel uses a single `BzlLoadFunction` Skyframe function with **context-specific SkyKey types** to handle all .bzl loading uniformly:

| Context | Key Type | Repo Mapping | Restrictions |
|---|---|---|---|
| `load()` in BUILD file | `KeyForBuild` | Full bzlmod mapping | None |
| `load()` in WORKSPACE | `KeyForWorkspace` | WORKSPACE-scoped | Only up to current WORKSPACE chunk |
| `load()` in module extension .bzl | `KeyForBzlmod` | Full bzlmod mapping | None |
| `@bazel_tools` during bootstrap | `KeyForBzlmodBootstrap` | Only `@bazel_tools` | Avoids cycles |
| `@_builtins` pseudo-repo | `KeyForBuiltins` | Only `@_builtins` | No builtins injection |

**Source**: [`BzlLoadFunction.java`](https://github.com/bazelbuild/bazel/blob/master/src/main/java/com/google/devtools/build/lib/skyframe/BzlLoadFunction.java), [`BzlLoadValue.java`](https://github.com/bazelbuild/bazel/blob/master/src/main/java/com/google/devtools/build/lib/skyframe/BzlLoadValue.java)

#### Bazel's Bootstrapping Solution

Bazel **never** needs to execute extensions before Skyframe starts. Instead:

1. **`BazelDepGraphFunction`** processes all `MODULE.bazel` files and their `use_extension()`/`use_repo()` declarations. From this alone—**without running any extensions**—it generates **deterministic canonical names** using:
   ```
   canonical_name = @<module_name>+<extension_name>+<repo_name>
   ```
   (via `makeUniqueNameCandidate()`)

2. **`RepositoryMappingFunction`** builds `apparent_name → canonical_name` mappings for each repo by depending on `ModuleExtensionRepoMappingEntriesFunction`, which in turn depends on `SingleExtensionFunction`.

3. **`SingleExtensionEvalFunction`** (the actual extension execution) runs **lazily inside Skyframe**, triggered only when a repo generated by that extension is actually referenced. It checks three lockfile digests first:
   - `bzlTransitiveDigest` — hash of extension .bzl + all transitive loads
   - `usagesDigest` — hash of all tags from all modules
   - `recordedInputs` — file/env digests

4. If lockfile is valid, cached `generatedRepoSpecs` are used directly. No .bzl loading needed.

**Source**: [`SingleExtensionEvalFunction.java`](https://github.com/bazelbuild/bazel/blob/master/src/main/java/com/google/devtools/build/lib/bazel/bzlmod/SingleExtensionEvalFunction.java), [`BazelDepGraphFunction.java`](https://github.com/bazelbuild/bazel/blob/master/src/main/java/com/google/devtools/build/lib/bazel/bzlmod/BazelDepGraphFunction.java)

#### Bazel's repository_rule() Execution

When a `repository_rule()` is invoked from a module extension:
1. The call is **deferred** into a `LinkedHashMap<String, RepoRuleCall>` via `ModuleExtensionEvalStarlarkThreadContext.lazilyCreateRepo()`
2. After the extension body returns, `createRepos()` invokes `repoRule.instantiate()` on each deferred call, producing `Map<String, RepoSpec>`
3. Actual fetching (disk I/O) happens later via `RepositoryDelegatorFunction` → `StarlarkRepositoryFunction`, triggered lazily when the repo's content is first needed

**Source**: [`ModuleExtensionEvalStarlarkThreadContext.java`](https://github.com/bazelbuild/bazel/blob/master/src/main/java/com/google/devtools/build/lib/bazel/bzlmod/ModuleExtensionEvalStarlarkThreadContext.java), [`StarlarkRepositoryModule.java`](https://github.com/bazelbuild/bazel/blob/master/src/main/java/com/google/devtools/build/lib/bazel/repository/starlark/StarlarkRepositoryModule.java)

### 2. Buck2's Approach: No bzlmod, No repository_rule()

Buck2 has **no equivalent** to Bazel's bzlmod, MODULE.bazel, WORKSPACE, `repository_rule()`, or module extensions.

| Feature | Bazel | Buck2 | Slug |
|---|---|---|---|
| External dep declaration | `MODULE.bazel` + `bazel_dep()` | `.buckconfig [cells]` | `MODULE.bazel` (added) |
| Custom fetch logic | `repository_rule()` in Starlark | None | `repository_rule()` (added) |
| Module extensions | `module_extension()` | None | `module_extension()` (added) |
| Pre-build Starlark phase | Skyframe-based extension eval | None | Sync executor + DICE executor (added) |
| External cell origin types | N/A (repos are the model) | `Bundled`, `Git` (2 variants) | `Bundled`, `Git`, `LocalPath`, `Bzlmod`, `RepositoryRule`, `ExtensionRepo` (6 variants) |

Buck2's external dependencies are either:
- **Vendored** with hand-written BUCK files
- Fetched via `http_archive` / `git_fetch` as **regular build actions** (not pre-build)
- Managed by external tools like Reindeer (for Rust/Cargo)

The cell system is **static**: cells are directories declared in `.buckconfig`, resolved before DICE, and injected as an `InjectedKey`. External cells can be `Bundled` (baked into binary) or `Git` (specific commit hash). There is no Starlark hook for custom fetch logic.

**Source**: [Buck2 External Cells docs](https://buck2.build/docs/users/advanced/external_cells/), [Issue #132](https://github.com/facebook/buck2/issues/132)

Buck2's DICE bootstrapping is straightforward because there's no dynamic repo discovery:
1. Read `.buckconfig` → build `CellResolver` (pre-DICE, synchronous)
2. Inject `CellResolver` into DICE as `InjectedKey`
3. Per-cell configs computed lazily inside DICE
4. External cell content materialized lazily inside DICE via `ExternalCellsImpl`

No chicken-and-egg problem because cell names are always statically known.

### 3. Slug's Four .bzl Loading Paths

Slug has **four** distinct mechanisms for loading and evaluating `.bzl` files, each with different capabilities:

#### Path 1: Normal DICE-Based Loading (BUILD files, regular .bzl)

- **Entry**: `InterpreterCalculation::get_loaded_module()` → `EvalImportKey::compute()` → `DiceCalculationDelegate::eval_module_uncached()`
- **Globals**: Full `GlobalInterpreterState` via `base_globals()` — includes `rule()`, `aspect()`, `provider()`, `select()`, `host_info()`, `native.*`, all 17 Starlark stdlib extensions, rules_cc autoloading, prelude injection
- **Load resolution**: Full cell resolver + `CellPath` via `InterpreterLoadResolver::resolve_load()`
- **Module cache**: DICE key (`EvalImportKey`) — globally cached across the build
- **File I/O**: `DiceFileComputations::read_file()`

#### Path 2: Module Extension Executor (DICE-based async)

- **Entry**: `ConcreteModuleExtensionExecutor::execute_extension()` in `module_extension_executor_impl.rs`
- **Globals**: The `.bzl` file is loaded through Path 1 (full globals). The `implementation(module_ctx)` call uses a bare `Evaluator::new()` — no custom globals, but the implementation closure has access to its frozen module's scope
- **Load resolution**: Via DICE (full cell resolver)
- **How it loads**: `parse_bzlmod_bzl_path()` → `ImportPath` → `ctx.get_loaded_module(StarlarkModulePath::LoadFile(...))`

#### Path 3: Starlark Repo Rule Executor (DICE-based async)

- **Entry**: `ConcreteStarlarkRepoRuleExecutor::execute_rule()` in `starlark_repo_rule_executor_impl.rs`
- **Globals**: Same as Path 2 — .bzl loaded through DICE, implementation called with bare evaluator
- **Load resolution**: Same `parse_bzlmod_bzl_path()` → DICE path
- **Triggered from**: `ExtensionRepoExecutionKey::compute()` for non-builtin rules

#### Path 4: Sync Extension Executor (pre-DICE, `DiskFileLoader`)

- **Entry**: `ConcreteSyncExtensionExecutor::execute_extension_sync()` in `sync_extension_executor_impl.rs`
- **Triggered from**: `try_execute_extension_sync()` in `cells.rs` when an extension is NOT in the lockfile cache
- **Globals**: Hand-built `build_extension_globals()` — only `repository_rule()`, `module_extension()`, `attr.*`, `register_bzl_module_globals()`, `Print`, `Pprint`
- **Load resolution**: `DiskFileLoader::resolve_path()` — filesystem path math against `project_root`
- **Module cache**: `Mutex<HashMap<String, FrozenModule>>` — process-local, non-shared
- **File I/O**: `std::fs::read_to_string()`
- **Key limitation**: `@cell//...` loads return `None` (cannot resolve external cells)

### 4. Inconsistencies Between Sync Executor and Bazel/Normal Paths

#### 4a. Missing Globals in Sync Executor

The sync executor's `build_extension_globals()` registers ~4 function groups vs ~20+ in the normal path:

| Available in Sync | Available in DICE | Missing from Sync |
|---|---|---|
| `repository_rule()` | `repository_rule()` | |
| `module_extension()` | `module_extension()` | |
| `attr.*` | `attr.*` | |
| `register_bzl_module_globals()` | `register_bzl_module_globals()` | |
| `Print`, `Pprint` | `Print`, `Pprint` | |
| — | `rule()` | `rule()` |
| — | `aspect()` | `aspect()` |
| — | `select()` | `select()` |
| — | `host_info()` | `host_info()` |
| — | `provider()` | `provider()` |
| — | `read_config()` | `read_config()` |
| — | `native.*` namespace | `native.*` namespace |
| — | All 17 stdlib extensions (Json, Map, Filter, EnumType, RecordType, StructType, Typing, etc.) | All 17 stdlib extensions |
| — | `cfg()`, transition globals | `cfg()`, transition globals |
| — | BXL globals | BXL globals |

This means that any extension `.bzl` file that uses `select()`, `host_info()`, `json.decode()`, `struct()`, or any stdlib function will **silently fail** in the sync path.

#### 4b. No External Cell Load Resolution

`DiskFileLoader` at `sync_extension_executor_impl.rs:129-132`:
```rust
} else if path.starts_with('@') {
    tracing::debug!("Cannot resolve external load '{}' in sync mode", path);
    None
}
```

Any `load("@some_dep//pkg:file.bzl", ...)` silently fails. This means extensions defined in external modules (which is the common case for `rules_python`, `rules_oci`, etc.) cannot be executed in sync mode.

#### 4c. Repo Name Resolution Collapse

`resolve_bzl_to_fs_path()` at `sync_extension_executor_impl.rs:374-380` strips the `@@repo` prefix and resolves everything against the project root:
```rust
let path_without_at = if bzl_path.starts_with("@@") {
    bzl_path.strip_prefix("@@").unwrap()
} else ...
```

This means `@@rules_python//pip:pip.bzl` and `@@my_custom//pip:pip.bzl` would both resolve to `{project_root}/pip/pip.bzl` — different repos collapse to the same filesystem path. Only root-module extensions (`//:ext.bzl`) work correctly.

#### 4d. Thread-Local State vs DICE Context

The sync executor uses `set_bzl_context()` (a thread-local in `repo_spec.rs`) to track the current `.bzl` file being evaluated, so that `repository_rule()` invocations know their source path for building `repo_rule_id`. This is a process-global side-effect pattern that doesn't compose with concurrent evaluation. The DICE-based paths instead derive this from the module path context.

#### 4e. Module Caching Divergence

- **Normal DICE path**: `EvalImportKey` is a DICE key — modules are cached globally across the entire build graph with change tracking
- **Sync executor**: `Mutex<HashMap<String, FrozenModule>>` — per-executor-invocation, discarded after execution, no invalidation tracking

This means the same `.bzl` file will be evaluated twice if accessed first in sync mode, then later via DICE.

#### 4f. `BuildContext` / Extra Starlark Context Not Available

The sync evaluator does not set `eval.extra` (the `BuildContext` that the normal path injects). This means any `.bzl` code that accesses `BuildContext::from_context()` will fail or get a fallback value. The recent changes made these checks optional, but the behavior diverges.

### 5. How Bazel Avoids This Problem Entirely

Bazel's architecture avoids a sync executor because:

1. **Canonical names are pre-computed**: `BazelDepGraphFunction` generates repo canonical names deterministically from the module graph topology alone — no extension execution needed.

2. **Extension execution is lazy inside Skyframe**: `SingleExtensionEvalFunction` runs inside the Skyframe DAG. When a repo from an extension is first referenced, Skyframe requests the `SINGLE_EXTENSION_EVAL` node, which may trigger `.bzl` loading (also inside Skyframe). If the lockfile cache is valid, no execution happens at all.

3. **Skyframe handles ordering automatically**: If a BUILD file references `@pip~numpy`, Skyframe's `RepositoryMappingFunction` depends on `SingleExtensionFunction` for pip, which depends on `SingleExtensionEvalFunction`. If the eval isn't ready, Skyframe returns `null` and restarts the requesting computation later. No explicit synchronization needed.

4. **The lockfile provides a fast path**: On subsequent builds, `SingleExtensionEvalFunction` short-circuits using cached `generatedRepoSpecs` without loading any `.bzl` files.

The critical difference: Bazel **does not need repo names before Skyframe starts** because Skyframe can lazily compute them. Slug needs repo names before DICE starts because Buck2's cell system was designed around static `.buckconfig` declarations.

### 6. Potential Alternatives to the Current Implementation

#### Alternative A: Defer Cell Registration to Inside DICE

Instead of registering extension repo cells before DICE starts, make the cell resolver itself lazy:

- Register a **placeholder cell** (like `_main~ext~repo`) with an `ExtensionRepo` origin that includes the `AggregatedExtension` data
- When the cell is first accessed inside DICE, compute the `ModuleExtensionExecutionKey` to run the extension via the full DICE path
- This eliminates the sync executor entirely

**Trade-off**: Requires making the cell resolver mutable or using a two-phase cell resolver (initial + lazy extensions). The current Buck2 `CellResolver` is immutable after construction. This would be a deeper architectural change but would align with Bazel's Skyframe approach.

#### Alternative B: Lockfile-Only for Pre-DICE, DICE for First Build

- **If lockfile exists and is valid**: Use cached `generatedRepoSpecs` to register cells (current behavior, works well)
- **If lockfile is missing**: Instead of the sync executor, register cells optimistically using the pre-computed canonical name scheme (`_main~{ext}~{repo}`) based solely on `use_repo()` declarations (which are known from MODULE.bazel parsing). Mark these cells as "unresolved"
- **First build access**: When an unresolved cell is accessed, trigger extension execution inside DICE

**Trade-off**: Requires that `use_repo()` names match what the extension actually generates. This is the approach Bazel takes — `BazelDepGraphFunction` pre-computes names, and `SingleExtensionFunction` validates them.

#### Alternative C: Make the Sync Executor More Complete

Keep the sync executor but add the missing globals and improve load resolution:

- Add all stdlib extensions (`Json`, `Map`, `Filter`, etc.) to `build_extension_globals()`
- Add `select()`, `host_info()`, `struct()`, etc.
- Add support for `@cell//...` loads by reading already-fetched external cells from `bazel-external/` or `~/.cache/slug/registry/`

**Trade-off**: Creates and maintains two parallel Starlark environments. Every new global added to the normal path must also be added to the sync path. This is the current trajectory and has scaling concerns.

#### Alternative D: Run DICE Early with a Bootstrap Cell Resolver

- Build an initial `CellResolver` with only statically-known cells (from `.buckconfig` + `bazel_dep()`)
- Start DICE with this initial resolver
- Execute extensions inside DICE (full `.bzl` loading works)
- Update the `CellResolver` with extension-generated cells (via `changed_to()` on the `CellResolverKey`)
- Recompute any DICE nodes that depended on the old resolver

**Trade-off**: DICE supports `InjectedKey` updates via `changed_to()`, but changing the cell resolver mid-computation could invalidate a large portion of the graph. This is essentially what Bazel's Skyframe does naturally — nodes restart when their dependencies change.

## Architecture Documentation

### Current Slug Extension Execution Pipeline

```
MODULE.bazel parsed
       │
       ▼
use_extension("//:ext.bzl", "my_ext")  ──→  AggregatedExtension built
use_repo(my_ext, "my_repo")
       │
       ▼
cells.rs: resolve_extension_repos_from_lockfile()
       │
       ├── Lockfile HIT ──→ Use cached generatedRepoSpecs ──→ Register cells
       │
       └── Lockfile MISS ──→ try_execute_extension_sync()
                                    │
                                    ├── Extension is root-local (//:ext.bzl)
                                    │   └── DiskFileLoader reads from disk
                                    │       └── Standalone Starlark evaluator
                                    │           └── Captures RepoSpecs
                                    │               └── Cache in lockfile
                                    │                   └── Register cells
                                    │
                                    └── Extension is external (@dep//ext.bzl)
                                        └── FAILS (cannot resolve @dep)
                                            └── Cell not registered
                                                └── "unknown cell" error

DICE starts with registered cells
       │
       ▼
@my_repo referenced in BUILD ──→ ExtensionRepoExecutionKey::compute()
       │
       ├── Builtin rule (http_archive, etc.) ──→ execute_repository_rule() native
       │
       └── Custom rule (//:rules.bzl%my_rule) ──→ StarlarkRepoRuleExecutor
                                                      │
                                                      └── Load .bzl via DICE (full path)
                                                          └── Execute rule.impl(repository_ctx)
```

### Bazel's Pipeline (for Comparison)

```
MODULE.bazel parsed
       │
       ▼
BazelDepGraphFunction: Pre-compute ALL canonical names
  (NO extension execution, just deterministic naming)
       │
       ▼
Skyframe starts with full canonical name mappings
       │
       ▼
@my_repo referenced in BUILD
       │
       ▼
RepositoryMappingFunction: apparent → canonical
       │ (depends on SingleExtensionFunction)
       │
       ▼
SingleExtensionEvalFunction:
       │
       ├── Lockfile HIT ──→ Return cached generatedRepoSpecs
       │
       └── Lockfile MISS ──→ Load extension .bzl via BzlLoadFunction(KeyForBzlmod)
                              └── Execute extension impl
                                  └── Deferred repo rule calls → RepoSpec map
                                      └── Cache in lockfile
       │
       ▼
RepositoryDelegatorFunction: Lazy repo materialization
       │
       └── StarlarkRepositoryFunction: Execute repo rule impl on disk
```

## Open Questions

1. **Should Slug adopt Bazel's pre-computed canonical naming?** This would let us register placeholder cells from `use_repo()` declarations alone, eliminating the need for sync execution entirely. Requires that `use_repo()` names match what extensions generate (Bazel enforces this via `SingleExtensionFunction` validation).

2. **Can the `CellResolver` be updated inside DICE?** Buck2's `CellResolverKey` is an `InjectedKey` that can be updated via `changed_to()`. If extension execution happens inside DICE and updates the resolver, downstream nodes would be invalidated and recomputed. Is this practical with DICE's invalidation model?

3. **What happens when an external-cell extension isn't in the lockfile?** Currently: silent failure and "unknown cell" error. Bazel would lazily execute the extension inside Skyframe. The gap between these behaviors is where the sync executor was added, but it only covers root-project extensions.
