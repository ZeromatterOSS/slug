## Section 3: Triage Suggestions

### Consumer 1: `read_config()` / `read_root_config()` — **KEEP**

All 84 call sites use `read_config` as a **test coordination mechanism** (passing per-test parameters via `-c section.key=value`). This is not a user-facing buckconfig feature — it is essential infrastructure for the test framework itself. The `read_config()` Starlark function must remain. None of these fixtures are "testing buckconfig reading as a feature"; they use it as a transport layer.

**Flag for special attention:** The 4 `read_root_config` callers reference `core_test_errors/broken_select_in_toolchain`, `test/buck2_output`, and `nested/buck2_path`. The `nested/buck2_path` and `test/buck2_output` keys suggest Buck2-specific naming that may need renaming. Low priority.

### Consumer 2+3: `config_setting` buckconfig-dotted-key path — **DEAD CODE PATH in tests**

Zero fixtures exercise the `section.property` dotted-key format in `values =`. All current `config_setting(values=...)` in tests use Bazel well-known keys (`compilation_mode`, `cpu`) which take a different code path. The dotted-key branch in `calculation.rs` (line 461–478) and `configuration.rs` (line 102–117) has no test coverage.

**Triage:** The `test_select_config_setting_data/BUILD.bazel` fixtures test the Bazel well-known key path and are worth keeping. No fixture deletion needed. The dead dotted-key code path in the Rust sources is a separate question (not in scope here).

### Consumer 4: `error_on_dep_only_incompatible` — **3 tests, purpose-built**

All 3 affected test functions (`test_error_on_dep_only_incompatible`, `test_error_on_dep_only_incompatible_conf`, `test_error_on_dep_only_incompatible_excluded`) exist specifically to test this feature. The feature is in `nodes.rs`. If the knob is being deleted, these 3 tests should be deleted too (or migrated to whatever replaces the feature).

The `.buckconfig` entry in `test_target_incompatible_data/.buckconfig` would also need removal.

### Consumer 5: `allow_eden_io` — **INCIDENTAL, safe to delete entries**

The 2 `.buckconfig` entries setting `allow_eden_io = false` are defensive guards against Eden IO in CI. Since the code is `#[cfg(fbcode_build)]`-gated (dead in OSS), these entries are functionally inert. The tests they belong to (`test_errors_data`, `test_uncategorized_data/artifact_consistency`) are not testing Eden IO — `allow_eden_io` is incidental. The `.buckconfig` lines can be deleted without affecting test behavior.

### Consumer 6: `[buildfile]` — **ESSENTIAL, do not touch**

267 fixture `.buckconfig` files declare `[buildfile] name = TARGETS.fixture` (or similar) so kuro can find the fixture build files. This is the mechanism that allows `TARGETS.fixture` naming. If these are being migrated to `BUILD.bazel`, the `[buildfile]` entries would be removed as part of that rename (Plan 35.6b task 12). Until then: do not touch.

### Consumer 7: `init.rs` — **DEAD IMPORT, trivial cleanup**

`use crate::legacy_configs::key::BuckconfigKeyRef;` at line 21 is unused. Remove it (compiler may warn). Zero test impact.
