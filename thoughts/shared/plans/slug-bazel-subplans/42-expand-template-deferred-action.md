## Context

Plan 03 marks `ctx.actions.expand_template` as **Done** but with a
known limitation: the implementation reads the template synchronously
at analysis time via `std::fs::read_to_string`. That works for
source-file templates (slug's main exercised case via llvm-project
and rules_python) but rejects build-artifact templates with:

```
expand_template: template artifact has no source path
```

ZeroMatter `//sdk:sdk_contents` hits this through `bazel_lib`'s
`expand_template` rule:

```python
def _expand_template_impl(ctx):
    ctx.actions.expand_template(
        template = ctx.file.template,   # <-- generated by another rule
        output = output,
        substitutions = substitutions,
        is_executable = ctx.attr.is_executable,
    )
```

`ctx.file.template` resolves to a `BuildArtifact` here; `source_path_info()`
returns `None`; slug errors out. The TODO at
`app/slug_action_impl/src/context/write.rs:698-702` calls this out
explicitly:

```rust
// TODO(slug): This reads templates directly from disk at analysis time,
// bypassing DICE/artifact infrastructure. Replace with an action or
// proper source-file materialization that goes through the normal
// artifact path.
```

## Design

Promote `expand_template` from a synchronous "read + write" at
analysis time to a real `Action` that runs at execution time, with
the template declared as an action input.

### New module `actions/impls/expand_template.rs`

Mirrors `actions/impls/copy.rs` (single-input, single-output, simple
deferred action). Three types:

- `UnregisteredExpandTemplateAction`: holds `template: ArtifactGroup`,
  `substitutions: Vec<(String, String)>`, `is_executable: bool`.
- `ExpandTemplateAction` (built via `register()`): owns the inputs
  (template) and the output (single `BuildArtifact`).
- `Action::execute` resolves the template's value via
  `ctx.artifact_values(template)`, reads its bytes through the
  artifact filesystem, applies substitutions, writes via
  `ctx.materializer().declare_write(...)`.

Substitution semantics match Bazel: each key replaced as-is in the
template content. No regex; literal substring. Order of substitutions
follows the dict iteration order at analysis time.

### Starlark glue (`context/write.rs::expand_template`)

Replace the synchronous read path with:

1. Resolve the `template` value to an `ArtifactGroup`. Source files
   become `ArtifactGroup::Artifact(<source artifact>)`; build
   artifacts likewise become `ArtifactGroup::Artifact(<build
   artifact>)`. Either is valid input.
2. Collect substitutions from the dict / `template_dict` argument
   into a `Vec<(String, String)>` *and* their associated artifacts
   (when `template_dict` is involved, its `add_joined` may reference
   command-line args that need to be flattened first — keep the
   existing flattening, just on the analysis side).
3. Build `UnregisteredExpandTemplateAction` and register via
   `state.register_action`.

Source-file templates flow through the same code path; the action
runs but reads a source artifact (which is essentially free — no
upstream DAG to wait on).

### Registration

Add `pub(crate) mod expand_template;` to `actions/impls.rs`.

## Out of scope

- Full Bazel `template_dict` support beyond what the analysis path
  already handles. The current Starlark-side flattening can keep
  working.
- Computed substitutions (slug already supports these via
  `StarlarkTemplateDict`).
- Generalising the "read at analysis time" anti-pattern to other
  `ctx.actions.*` callers — that's a follow-up plan if a new symptom
  surfaces. Today's known sites are limited to expand_template.
