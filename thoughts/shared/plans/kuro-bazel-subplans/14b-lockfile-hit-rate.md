# Plan 14b: Fix Lockfile Hit-Rate for Extension Cache

> Parent: Plan 10 Phase 7 follow-up
>
> Note: Original Option 2 ("parallelize load() resolution") is already
> implemented at `app/kuro_interpreter_for_build/src/interpreter/dice_calculation_delegate.rs:218`
> via `eval_deps` + `try_compute_join`. Substituting with lockfile hit-rate,
> which addresses the same symptom (serial-looking extension cascade) from
> a different angle: make extension re-execution unnecessary by hitting
> the lockfile cache.

## Problem

`ModuleExtensionExecutionKey::compute` checks the lockfile cache before
running extensions:

```rust
if let Some(cached_specs) = lockfile.get_extension_cache(
    &self.extension_id,
    &bzl_transitive_digest,
    &usages_digest,
) {
    return Ok(Arc::new(result));
}
```

But `compute_bzl_transitive_digest` is stubbed
(`extension_execution_dice.rs:703-717`): it hashes only the extension ID,
not the actual `.bzl` file contents. This means the digest is stable
across runs — cache *should* hit. Yet empirical observation of
`@llvm-project//llvm:config` cquery shows many "triggering lazy
execution" logs, indicating cache misses.

Observed log line: `WARN ... Extension cache digest mismatch for
'//:extensions.bzl%llvm_repos_extension' (using lockfile specs anyway)`.
Some extensions *do* fall through to use lockfile data despite the
mismatch; most do not. This is wasted work: re-executing extensions
whose results are already stored.

## Bazel Parity

Bazel's extension lockfile records:
- `extensionId` (with usage hash)
- `bzlTransitiveDigest` (SHA256 of all .bzl files transitively loaded by
  the extension's .bzl, read from the file contents as they existed
  during the previous successful build)
- `usagesDigest` (SHA256 of all extension usages, including tag classes,
  attribute values, and the consuming modules' names/versions)
- `generatedRepoSpecs` (the captured RepoSpec objects)

On re-execution, Bazel compares current digests against the lockfile's
stored digests. If both match, it skips execution entirely. `.bzl`
digests are read via `BzlLoadFunction` and reflect the actual file bytes
as of the current build. If `.bzl` content hasn't changed, digests match
and the lockfile hits.

Kuro's current impl simply doesn't compute real digests, so:
- `compute_bzl_transitive_digest` hashes `extension_id` — stable but
  doesn't reflect .bzl content changes. If lockfile was written with the
  same stubbed digest, should hit.
- `usages_digest` comes from `input_hash` → `compute_extension_input_hash`
  — reads actual tag data, seems real.

The mismatch is likely in `bzl_transitive_digest` not matching between
current-run and lockfile-stored. Either:
1. Lockfile stores a different digest format than current runs compute.
2. Lockfile stores real Bazel digests (from a Bazel-generated
   `MODULE.bazel.lock`) and kuro's stubbed digest doesn't match.

The lockfile for llvm-project/utils/bazel was generated by Bazel and
contains Bazel-computed digests. Kuro's stub cannot match.

## Design

Two-step fix:

### Step 1: Lenient digest matching for Bazel-generated lockfiles

When the stored digest doesn't match the current-run digest but the
usages_digest matches exactly, trust the lockfile specs. Matches the
"using lockfile specs anyway" path that already exists for
`llvm_repos_extension` — generalize this behavior.

Location: `extension_execution_dice.rs:406` lockfile read block.

### Step 2: Implement real `bzl_transitive_digest`

Replace the stub in `compute_bzl_transitive_digest`
(`extension_execution_dice.rs:703-717`) with a real implementation that:

1. Loads the extension's `.bzl` module via DICE
   (`get_loaded_module_imports` already returns transitive deps)
2. For each transitively loaded `.bzl` file, reads file content via DICE
3. Hashes all contents together with SHA256, SRI-formatted

This matches Bazel's `BzlTransitiveDigestUtil.getDigest()`.

Once step 2 lands, step 1's lenient matching becomes unnecessary — digests
will agree when content agrees. Keep step 1 as a short-term fix while
step 2 is being implemented and tested.

## Touchpoints

- `app/kuro_bzlmod/src/extension_execution_dice.rs:406` — lockfile read
  block: accept specs when `usages_digest` matches even if
  `bzl_transitive_digest` doesn't (behind a feature flag / flag in the
  lockfile header, so strict matching can be re-enabled when step 2 is
  ready)
- `app/kuro_bzlmod/src/extension_execution_dice.rs:703-717` — replace
  stubbed `compute_bzl_transitive_digest`:
  - Change signature to take `&mut DiceComputations<'_>` so it can use
    DICE to load .bzl imports
  - Find the extension's parent `.bzl` via parsed extension_id
  - Use `get_loaded_module_imports` to get transitive imports
  - Read file content for each via `DiceFileComputations::read_file`
  - Hash in deterministic order (sort by path)
- `app/kuro_bzlmod/src/lockfile.rs` — if the stored format differs from
  Bazel's, add a shim that reads Bazel's format too (it's JSON,
  straightforward)

## Risks

1. **Digest format divergence from Bazel**: Bazel's lockfile is a moving
   target. If kuro computes digests differently (e.g., different hash
   algorithm, different serialization), every lockfile miss persists.
   Mitigation: Bazel uses SHA256 + SRI encoding, documented in
   [bazelbuild/bazel DigestUtil](https://github.com/bazelbuild/bazel/blob/master/src/main/java/com/google/devtools/build/lib/bazel/bzlmod/BzlmodUtil.java).
   Replicate exactly. Reference
   `app/kuro_bzlmod/src/lockfile.rs` existing SRI handling.

2. **Real digest compute requires DICE access** — the current function
   is a pure helper. Restructuring to take DICE means caller sites
   change. Mitigation: straightforward; only one caller
   (`ModuleExtensionExecutionKey::compute`) and it already has `ctx`.

3. **Transitively loaded .bzl files may include extension-generated
   files** — creates a cycle: to compute digest for extension X, must
   load files from cells generated by extension X. Mitigation: for
   digest purposes, treat missing/pending cells as "digest not
   computable" and fall back to usages_digest-only match (step 1
   behavior). This is a graceful degradation.

4. **Partial lockfile data**: an entry may be present but incomplete
   (e.g., missing some repo specs due to a previous failed-halfway
   build). Mitigation: `get_extension_cache` already returns
   `Option<HashMap>`. If the loaded set doesn't have the requested repo,
   fall through to execution. Already handled at `extension_repo.rs:580`.

## Not Addressed

- Extensions that legitimately need fresh execution (e.g., the
  implementation changed in .bzl): those must execute and populate the
  cache. Nothing to fix.
- Cases where .bzl content didn't change but the extension's external
  inputs did (HTTP archive URL changed, etc.): Bazel's lockfile also
  doesn't handle this perfectly — both rely on the user to invalidate.
